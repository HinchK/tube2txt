Architecture Overview

Gridland TUI + FastAPI WebSocket Backend — Single Container

System Architecture
GRIDLAND TUI (Bun)
Screens
  ProcessScreen — URL input, mode toggles, live log
  DashboardScreen — video grid, metadata cards
  VideoDetailScreen — inline transcript + AI content
  SearchScreen — FTS5 query + results
State
  useWebSocket() — connection + message handling
  useVideos() — REST data fetching
  useSearch() — search with debounce
⟵ REST ⟶
⟵ WS ⟶
FASTAPI BACKEND (Python)
REST Endpoints
  GET  /api/videos — list all videos
  GET  /api/videos/{slug} — video detail + segments
  GET  /api/search?q= — FTS5 search
  GET  /api/videos/{slug}/ai — AI content files

Domain Layer (existing modules)
  Downloader → VTTParser → GeminiClient
  HTMLGenerator → Database → ClippingEngine
tube2txt.db (SQLite + FTS5)
projects/ (generated artifacts)

Key Design Decisions

  • hub.py transforms from a full HTML-serving app into a headless API — existing Database, search, and static file logic stays, HTML rendering is stripped
  • process_video_logic gets wrapped with progress callbacks that push messages over the WebSocket (download %, extraction status, AI generation steps)
  • Gridland app lives in a new tui/ directory at project root — separate Bun project with its own package.json
  • Single Docker image uses multi-stage build: Bun builds the Gridland app, Python stage runs FastAPI + serves the built TUI assets
